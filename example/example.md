# Client Integration Examples

This file shows the minimal shape for wiring artifacts generated by
`eth-genesis-generator` into common local devnet clients.

Pick one execution client, one beacon node, and one validator client:

- Execution: Geth or Reth.
- Beacon node: Prysm or Lighthouse.
- Validator client: Prysm or Lighthouse.

The commands are local single-node templates. Adjust paths, ports, peer
settings, and client versions for real multi-node networks.

## Generate Artifacts

```bash
eth-genesis-generator init --out ./devnet
# Optional: edit ./devnet/genesis.yaml before generating.
eth-genesis-generator generate \
  --config ./devnet/genesis.yaml \
  --out ./devnet/artifacts
```

The generated directory contains:

- `execution/genesis.json`
- `consensus/config.yaml`
- `consensus/genesis.ssz`
- `consensus/genesis.json`
- `validators/mnemonics.yaml`
- `validators/keystores/*.json`
- `validators/keystores/password.txt`
- `manifest.json`

`validators/mnemonics.yaml` and `validators/keystores/` contain validator
secret material. Do not commit them unless they are explicit fixtures for a
test.

## Docker Compose: Reth + Lighthouse

The compose example builds the current repository image, writes
`./devnet/genesis.yaml`, generates artifacts into `./devnet/artifacts`, prepares
Reth/Lighthouse runtime data under `./devnet/runtime`, then starts a local
Reth + Lighthouse devnet.

Run the genesis compose file first:

```bash
docker compose -f example/docker-compose.genesis.yml up --build --abort-on-container-exit
docker compose -f example/docker-compose.genesis.yml down
```

Then run the initialization compose file:

```bash
docker compose -f example/docker-compose.init.yml up --abort-on-container-exit
docker compose -f example/docker-compose.init.yml down
```

Then start the nodes:

```bash
docker compose -f example/docker-compose.yml up
```

The genesis compose uses `init --force`, so it rewrites `./devnet/genesis.yaml`
with the default template before generating artifacts. If you change
`network.chain_id` in `genesis.yaml`, also update Reth's `--network-id` in
`example/docker-compose.yml`.

## Shared Local Setup

Use the same JWT secret for the execution client auth RPC and the consensus
client execution endpoint.

```bash
export ARTIFACTS="$PWD/devnet/artifacts"
export RUNDIR="$PWD/run"
export JWT="$RUNDIR/jwt.hex"
export CHAIN_ID=32382
export FEE_RECIPIENT=0x1000000000000000000000000000000000000001
export VALIDATOR_KEYS="$ARTIFACTS/validators/keystores"
export VALIDATOR_PASSWORD="$ARTIFACTS/validators/keystores/password.txt"

mkdir -p "$RUNDIR"
openssl rand -hex 32 | tr -d '\n' > "$JWT"
chmod 600 "$JWT"
```

If `network.chain_id` was changed in `genesis.yaml`, update `CHAIN_ID` to match.
`VALIDATOR_KEYS` and `VALIDATOR_PASSWORD` point at the generated EIP-2335
keystores and password file.

## Execution Client: Geth

Initialize the database from the generated execution genesis:

```bash
mkdir -p "$RUNDIR/geth"
geth init \
  --datadir "$RUNDIR/geth" \
  "$ARTIFACTS/execution/genesis.json"
```

Start Geth with the authenticated Engine API on `127.0.0.1:8551`:

```bash
geth \
  --datadir "$RUNDIR/geth" \
  --networkid "$CHAIN_ID" \
  --http \
  --http.addr 127.0.0.1 \
  --http.port 8545 \
  --http.api eth,net,web3,txpool \
  --authrpc.addr 127.0.0.1 \
  --authrpc.port 8551 \
  --authrpc.vhosts localhost \
  --authrpc.jwtsecret "$JWT" \
  --nodiscover \
  --syncmode full
```

For Docker or remote consensus clients, bind `--authrpc.addr` to the reachable
interface and set `--authrpc.vhosts` accordingly.

## Execution Client: Reth

Initialize the database from the generated execution genesis:

```bash
mkdir -p "$RUNDIR/reth"
reth init \
  --datadir "$RUNDIR/reth" \
  --chain "$ARTIFACTS/execution/genesis.json"
```

Start Reth with the authenticated Engine API on `127.0.0.1:8551`:

```bash
reth node \
  --datadir "$RUNDIR/reth" \
  --chain "$ARTIFACTS/execution/genesis.json" \
  --network-id "$CHAIN_ID" \
  --http \
  --http.addr 127.0.0.1 \
  --http.port 8545 \
  --http.api eth,net,web3,txpool \
  --authrpc.addr 127.0.0.1 \
  --authrpc.port 8551 \
  --authrpc.jwtsecret "$JWT" \
  --disable-discovery
```

If Reth cannot parse a future fork field from the generated genesis, upgrade to
a client build that supports the configured execution fork.

## Beacon Node: Prysm

Start Prysm against the execution client's Engine API:

```bash
beacon-chain \
  --datadir "$RUNDIR/prysm/beacon" \
  --chain-config-file "$ARTIFACTS/consensus/config.yaml" \
  --genesis-state "$ARTIFACTS/consensus/genesis.ssz" \
  --execution-endpoint http://127.0.0.1:8551 \
  --jwt-secret "$JWT" \
  --chain-id "$CHAIN_ID" \
  --network-id "$CHAIN_ID" \
  --suggested-fee-recipient "$FEE_RECIPIENT" \
  --min-sync-peers 0 \
  --p2p-max-peers 0 \
  --no-discovery \
  --interop-eth1data-votes \
  --accept-terms-of-use
```

The `--interop-eth1data-votes` flag is useful for this generator's v1 model,
where the genesis state is pre-filled with validators and the deposit contract
bytecode is not predeployed.

## Beacon Node: Lighthouse

Lighthouse reads custom network metadata from a testnet directory. Create one
from the generated consensus files:

```bash
export LIGHTHOUSE_TESTNET="$RUNDIR/lighthouse-testnet"

mkdir -p "$LIGHTHOUSE_TESTNET"
cp "$ARTIFACTS/consensus/config.yaml" "$LIGHTHOUSE_TESTNET/config.yaml"
cp "$ARTIFACTS/consensus/genesis.ssz" "$LIGHTHOUSE_TESTNET/genesis.ssz"
printf "0\n" > "$LIGHTHOUSE_TESTNET/deposit_contract_block.txt"
```

The deploy block is set to `0` for this generator's pre-filled devnet genesis.
If you deploy and use a real deposit contract, replace it with that deployment
block.

Start Lighthouse against the execution client's Engine API:

```bash
lighthouse beacon_node \
  --datadir "$RUNDIR/lighthouse/beacon" \
  --testnet-dir "$LIGHTHOUSE_TESTNET" \
  --execution-endpoint http://127.0.0.1:8551 \
  --execution-jwt "$JWT" \
  --http \
  --http-address 127.0.0.1 \
  --http-port 5052 \
  --target-peers 0 \
  --allow-insecure-genesis-sync \
  --suggested-fee-recipient "$FEE_RECIPIENT"
```

`--allow-insecure-genesis-sync` is intended for testing and should not be used
on mainnet.

## Validator Client: Prysm

Import the validator keystores into a Prysm wallet once:

```bash
validator accounts import \
  --keys-dir "$VALIDATOR_KEYS" \
  --wallet-dir "$RUNDIR/prysm/wallet" \
  --wallet-password-file "$VALIDATOR_PASSWORD" \
  --account-password-file "$VALIDATOR_PASSWORD" \
  --accept-terms-of-use
```

Start the Prysm validator client against the Prysm beacon node:

```bash
validator \
  --datadir "$RUNDIR/prysm/validator" \
  --wallet-dir "$RUNDIR/prysm/wallet" \
  --wallet-password-file "$VALIDATOR_PASSWORD" \
  --chain-config-file "$ARTIFACTS/consensus/config.yaml" \
  --beacon-rpc-provider 127.0.0.1:4000 \
  --beacon-rest-api-provider http://127.0.0.1:3500 \
  --suggested-fee-recipient "$FEE_RECIPIENT" \
  --accept-terms-of-use
```

Use the Prysm validator client with a Prysm beacon node unless you have tested
the gRPC/REST compatibility for your exact client versions.

## Validator Client: Lighthouse

Import the validator keystores into the Lighthouse validator directory once:

```bash
lighthouse account validator import \
  --datadir "$RUNDIR/lighthouse" \
  --testnet-dir "$LIGHTHOUSE_TESTNET" \
  --directory "$VALIDATOR_KEYS" \
  --password-file "$VALIDATOR_PASSWORD" \
  --reuse-password
```

Start the Lighthouse validator client against the Lighthouse beacon node:

```bash
lighthouse validator_client \
  --datadir "$RUNDIR/lighthouse" \
  --testnet-dir "$LIGHTHOUSE_TESTNET" \
  --beacon-nodes http://127.0.0.1:5052 \
  --suggested-fee-recipient "$FEE_RECIPIENT"
```

The same command can be written as `lighthouse vc`.

## Validator Key Material

Execution clients and beacon nodes can start without validator keys, but the
chain will not produce blocks until a validator client controls keys that match
the validators in `validators/mnemonics.yaml`.

`eth-genesis-generator` writes the mnemonic input used to create the genesis
validators and EIP-2335 keystores derived from the same `start` and `count`
values. Import those keystores into the selected validator client before
expecting the chain to advance.

Never run the same validator keys in more than one validator client at the same
time.
